# Project Structure

Understanding the layout of a tuvl project.

## Directory Layout

A typical tuvl project has the following structure:

```
my-project/
├── models/              # Model definitions (YAML)
│   ├── contact.yaml
│   └── order.yaml
├── workflows/           # Workflow definitions (YAML)
│   ├── contact_intake.yaml
│   └── order_processing.yaml
├── datasources/         # Database configurations (YAML)
│   └── postgres.yaml
├── agents/              # LLM agent presets (YAML)
│   └── default.yaml
├── nodes/               # Python node implementations
│   ├── __init__.py
│   ├── contact_nodes.py
│   └── order_nodes.py
├── .env                 # Environment variables (git-ignored)
├── .env.example         # Safe-to-commit template
└── .gitignore
```

## Core Directories

### `models/`

Contains ModelDefinition YAML files that define your data structures.

```yaml title="models/contact.yaml"
kind: "ModelDefinition"
version: "v1"
metadata:
  name: "Contact"
spec:
  tablename: "contacts"
  schema: true
  fields:
    - name: "id"
      type: "uuid"
      primary_key: true
    - name: "email"
      type: "string"
      required: true
```

Each model generates:

- A SQLModel class for database operations
- Pydantic schemas (`Create`, `Read`, `Update` variants)
- CRUD API endpoints at `/api/{model_name}`

### `workflows/`

Contains Workflow YAML files that define your business logic.

```yaml title="workflows/example.yaml"
kind: "Workflow"
metadata:
  name: "example_workflow"
trigger:
  path: "/api/example"
  method: "POST"
steps:
  - id: "step_1"
    kind: "functional"
    runner: "my_node"
```

### `datasources/`

Database connection configurations.

```yaml title="datasources/postgres.yaml"
kind: "DataSource"
version: "v1"
metadata:
  name: "main_postgres"
spec:
  type: "postgresql"
  driver: "asyncpg"
  connection:
    host: "${POSTGRES_HOST}"
    port: ${POSTGRES_PORT:5432}
    database: "${POSTGRES_DB}"
    username: "${POSTGRES_USER}"
    password: "${POSTGRES_PASSWORD}"
```

### `agents/`

LLM agent configurations and presets.

```yaml title="agents/default.yaml"
kind: "AgentModel"
version: "v1"
metadata:
  name: "default"
spec:
  provider: "ollama"
  model: "llama3"
  api_base: "http://localhost:11434"
  temperature: 0.7
  max_tokens: 1024
```

### `nodes/`

Python modules containing node implementations.

```python title="nodes/my_nodes.py"
from typing import Any
from tuvl_engine.nodes.base import node

@node("my_node")
async def my_node(ctx: dict[str, Any]) -> dict[str, Any]:
    # Your logic here
    return ctx
```

!!! important "Node Registration"
    Nodes must be imported at startup to register with the `NODE_REGISTRY`.
    Create an `__init__.py` that imports all node modules.

## Configuration Files

### `.env`

Environment variables for local development. **Never commit this file.**

```env
POSTGRES_HOST=localhost
POSTGRES_PORT=5432
POSTGRES_DB=tuvl
POSTGRES_USER=postgres
POSTGRES_PASSWORD=secret

OPENAI_API_KEY=sk-...
```

### `.env.example`

Template showing required variables with placeholder values. Safe to commit.

```env
POSTGRES_HOST=localhost
POSTGRES_PORT=5432
POSTGRES_DB=tuvl
POSTGRES_USER=postgres
POSTGRES_PASSWORD=

OPENAI_API_KEY=
```

### `.gitignore`

Generated by `tuvl init` with sensible defaults:

```gitignore
.env
__pycache__/
*.pyc
.venv/
```

## Runtime Directories

When running `tuvl dev`, the engine:

1. Loads all `*.yaml` files from `models/`
2. Generates SQLModel classes and schemas
3. Loads all `*.yaml` files from `workflows/`
4. Mounts HTTP routes for each workflow trigger
5. Imports Python files from `nodes/`
6. Registers decorated functions in `NODE_REGISTRY`

## Multi-Project Workspaces

For larger applications, you might organize multiple tuvl projects:

```
workspace/
├── services/
│   ├── crm/              # CRM service
│   │   ├── models/
│   │   ├── workflows/
│   │   └── nodes/
│   └── billing/          # Billing service
│       ├── models/
│       ├── workflows/
│       └── nodes/
├── shared/
│   └── nodes/            # Shared node implementations
└── docker-compose.yaml
```

Use the `--project-dir` flag to specify which project to run:

```bash
tuvl dev --project-dir services/crm
```
